# Tweaks panel writeback via Vite dev middleware

The tweaks panel needs to persist user changes from the in-page panel back to `src/data/tweak-defaults.ts` so that picking a hero variant or accent color sticks across reloads and triggers Vite HMR for structural tweaks (per ADR 0003). The pre-Astro implementation relied on the host page (a Claude Preview iframe) listening for `__edit_mode_set_keys` postMessages and rewriting the source file out-of-band; the `dev-server.py` deleted in slice 0001 was a static-file server and never participated in the writeback itself. With the migration to Astro, the panel can no longer assume any particular host environment, so the writeback has to live inside the project. We chose to add a Vite plugin (`src/dev/tweaks-writeback-plugin.ts`) that registers a `POST /__tweaks` middleware on the dev `connect` instance. The middleware reads the request body, locates the `/*EDITMODE-BEGIN*/ ... /*EDITMODE-END*/` region in the defaults file, and rewrites only that region preserving everything outside it byte-for-byte. The panel POSTs each mutation through a small debounce so rapid drag interactions on a segmented control coalesce into one write per knob.

## Considered Options

- **File System Access API** — modern Chromium browsers let a page write directly to a user-granted directory. Rejected: requires a one-time user gesture that re-prompts whenever the dev server restarts, only works in Chromium, and bakes a browser-API path into the panel that we'd need to special-case for unsupported browsers anyway. The Vite middleware path is uniform across every browser the panel might be open in.
- **"Copy snippet" UX pivot** — the panel stops persisting and shows a "copy as TweakDefaults" button the user pastes into source by hand. Rejected: the live-iteration loop the panel exists for (try a color, see the page, try another) becomes a manual copy-paste loop. We'd be paying the maintenance cost of the panel without delivering its core value.

## Consequences

- The dev server now owns a write-to-source endpoint. The endpoint is registered via `apply: "serve"` so the entire plugin is dropped from `astro build` — no `__tweaks` URL appears in the production bundle, no production code references the plugin, and the HTML emitted to `dist/` contains no trace of it (verified by spec 0009's acceptance criteria). The production deploy on GitHub Pages also serves only static files, so even if the URL leaked into the bundle it could not execute.
- The writeback parser/serializer is intentionally narrow: it expects the EDITMODE block to be a TS object literal of `key: "string-value",` pairs (matching what `tweak-defaults.ts` is constrained to today). Introducing a new field requires a real source edit; the panel cannot add keys, only update existing ones. Keys are validated against `/^[A-Za-z_][A-Za-z0-9_]*$/` and values against `/^[A-Za-z0-9_-]+$/`; anything outside the identifier-shape envelope (quotes, newlines, backslashes, …) is rejected with 422 so a hostile or buggy POST cannot inject syntax into the file. This keeps the plugin from having to model the full type system, and tracks what the underlying `TweakDefaults` string-literal unions already permit.
- A 120 ms debounce sits between the panel and the POST. This is long enough to coalesce a drag interaction (TweakRadio segments emit a steady stream of `onChange` calls while the pointer moves) and short enough that releasing the mouse persists within a frame or two. Concurrent POSTs are also serialized server-side via an async mutex so two near-simultaneous writes cannot interleave.
- The legacy `window.parent.postMessage({ type: "__edit_mode_set_keys", ... })` call is preserved alongside the new POST. Prototyping environments that embed the page in a parent capable of handling writeback themselves continue to work; the dev-server endpoint is just an additional channel that's wired by default in `astro dev`.
